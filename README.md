 🔔 Looker Alerting Extension

**Looker Alerting Extension** is a lightweight system that enables non-technical users to set up data quality checks and alerts in **Looker**, using only **SQL**—no LookML modeling required.

This project simplifies the process of creating alerts by using a Google Sheet as the control panel and automatically generating the necessary BigQuery views to power your Looker dashboards.

---

## 🚀 What Problem Does It Solve?

In most data monitoring setups, creating custom checks and alerts often requires LookML modeling or engineering support. We built this system to:

- Empower **non-technical users** to define their own data checks using SQL only.
- **Decouple alert logic** from Looker modeling.
- Enable fast iteration and testing of new alerts with minimal setup.

---

## ⚙️ How It Works

The system is made up of **three core components**:

1. **📝 Google Sheet** – acts as the *control center*, where users define queries.

2. **🗃️ BigQuery Tables & Views** – generated by the script based on the Google Sheet content. These contain the **results** of each check in a format consumable by Looker.

3. **📊 Looker Explore** – visual interface for creating **alerts** on top of the generated views.

**Workflow:**

- A script reads the Google Sheet and dynamically creates or updates the BigQuery views based on the provided SQL.
- Each view outputs standardized columns (`check_name`, `check_result`, `check_timestamp`) that are then connected to Looker.
- Users can set alerts directly in Looker using the Explore tied to these views.

---

## 🧱 Query Structure Requirements

All SQL checks must follow a standardized structure to allow the system to parse and aggregate results effectively.

```sql
WITH
source_table AS (
    SELECT
        col_1,
        col_2,
        col_3
    FROM
        tables
),

checks_table AS (
    SELECT
        "name_of_the_check"                AS check_name,
        source_table.col_3 > [[THRESHOLD]] AS check_result,
        CURRENT_TIMESTAMP()                AS check_timestamp
    FROM
        source_table
)
```

### ✅ Requirements Recap:

- Must contain a `source_` CTE and a `checks_` CTE.
- The `checks_` CTE should be the last piece of the query.
- CTE names must be **unique** per query and per Google Sheet.
- `checks_table` must include the following columns:
  - `check_name` (string)
  - `check_result` (boolean)
  - `check_timestamp` (timestamp)
- The `checks_` CTE must be built *on top of* the corresponding `source_` CTE.

---

## ⚠️ Current Limitations

- Only supports **boolean checks** (`True`/`False`).
- Only one check is allowed **per SQL query**.

## 📦 Getting Started

### Prerequisites:

- Google BigQuery access
- Looker instance with access to the BigQuery project
- Google Sheet API credentials (if automated ingestion is involved)

### Steps:

1. Clone this repo.
2. Create control Google Sheet (more in the templates folder).
3. Create Looker View and Explore (more in the templates folder).
4. Fill out the Google Sheet with your checks.
5. Run the notebook to generate / update BigQuery views.
6. Use the Looker Explore to configure alerts.
